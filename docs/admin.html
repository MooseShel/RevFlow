HIPAA Compliance & Security Audit Hub

Automated compliance analysis of technical safeguards, audit logging controls, and privacy-first logic implemented across the RevFlow infrastructure.

Access Control & RLS (§164.312(a)(1)) ▶
Technical Implementation Safeguards:
  • UUID v4 Tokenization: Links to patient statements use cryptographically random UUID v4 values. Statement records are completely decoupled from guessable linear primary keys.
  • PostgreSQL Row-Level Security (RLS): Access to the billing_statements and verification_tokens tables is disabled by default. The database session configures transaction-local variables (app.current_verified_token_id) that only allow records associated with the authenticated patient session to be selected.
  • Private Document Storage: Statements are archived in a locked storage bucket with private-only access permissions. The backend verification endpoint generates temporary secure signed URLs valid for exactly 1 hour.
Immutable Audit Logging (§164.312(b)) ▶
Forensic & Activity Audit Trail:
  • Comprehensive Logging: The access_audit_logs table records the complete lifecycle of every statement. Logs are permanent, append-only, and protected by RLS rules that prevent modification or deletion.
  • Traceable Events: Records GENERATED (on initial ingestion), VERIFIED (when identity is verified), VIEWED (when statement document is accessed), and ATTEMPT_FAIL (failed access attempts).
  • Forensic Metadata: Automatically captures client IP addresses, user agents, and token identifiers to support continuous compliance monitoring.
Data Integrity & Hashing (§164.312(c)(1)) ▶
Cryptographic Key Protections:
  • One-Way SHA-256 Hashing: Patient verification keys (ZIP Code and Phone number last 4 digits) are transformed into secure 256-bit hashes using Deno's native WebCrypto API before database insertion. Patient-supplied keys are matched strictly in their hashed state. Cleartext verification fields are never stored.
  • Social Security Numbers Omitted: Social Security Numbers are completely decoupled and excluded from the entire application layer. They are never processed by AI, stored in schemas, or rendered in UI structures.
Minimum Necessary PHI Dispatch (§164.502(b)) ▶
Notification Transmission Safeguards:
  • Zero PHI over SMS/Email: Notification messages dispatched via Resend Email and Twilio SMS strictly contain no Patient Health Information.
  • Exclusions: Subject lines and body text contain no patient names, clinical treatments, doctor details, or dollar amounts. Dispatches only contain a secure verification link and general notice that a statement is ready.
Logger PHI Redaction & Sanitization (§164.514) ▶
Application Log Sanitization:
  • Regex-Based Filter Rules: The central logger employs regular expressions to scan execution streams for sensitive data profiles (like full SSNs, phone numbers, or credit card formats).
  • Redaction Enforcement: Matches are replaced with [SSN REDACTED] or generic scrubbed values before logs are output to stdout/stderr, guaranteeing no PHI is written to developer logs.
